# Phase 04.1: Trip voting flow - Context

**Gathered:** 2026-04-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the trip creation flow so that after "Continue to Voting" in TripSheet, the user lands directly on the new trip's detail page ready to open voting. Unblocks manual UAT for Phase 4 and all future trip lifecycle testing.

This phase does NOT redesign the voting or confirmation flow — it fixes the navigation/creation path so the existing "Open voting" button in TripDetail is reachable.

</domain>

<decisions>
## Implementation Decisions

### Post-creation navigation
- **D-01:** After TripSheet creates a trip, navigate directly to `/trips/${docRef.id}` (TripDetail for the new trip) — NOT to `/trips` list. This matches how DirectTripSheet already works and immediately surfaces the "Open voting" button.

### Error handling
- **D-02:** Wrap the TripSheet submit handler in visible error feedback. If `addDoc` fails, show an inline error message (or toast using the project's existing pattern — check how other components surface errors). Do NOT silently swallow errors — the current `console.error` only is insufficient.

### Root cause investigation
- **D-03:** Before fixing navigation, verify whether the trip IS being created in Firestore (check browser console/Network tab or Firestore console). The investigation must determine:
  - Is `addDoc` throwing (permission error, validation)?
  - Is the `attendees` array being populated correctly (crew-load effect might be failing silently, leaving `attendees: []`)?
  - Is the Trips.tsx `onSnapshot` query failing due to a missing composite index?
- **D-04:** Even after the navigation fix (D-01), fix any underlying creation failure found in D-03 — navigating to an empty TripDetail would be equally broken.

### Attendees array guard
- **D-05:** Ensure the creator's own UID is always present in the `attendees` array regardless of crew selection. Add `creatorUid` explicitly to the array if not already present. This prevents the trip from being invisible in the Trips list and ensures the Firestore `array-contains` query matches.

### Scope
- **D-06:** Scope is limited to TripSheet's submit handler and its navigation/error feedback. Do NOT touch VotingPanel, closeVoting, confirmVoting, or any downstream lifecycle transitions — those are working.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `DirectTripSheet.tsx` — Already implements the correct post-creation navigation pattern (`navigate(\`/trips/${ref.id}\`)` then `onClose()`). The fix in TripSheet should mirror this exactly.
- Existing error UI patterns — check how other components (TripDetail, TripSheet form validation) surface errors to users. Use the same pattern for the creation failure toast.

### Established Patterns
- Trip status starts at `'proposed'` on creation — this is correct and should not change
- `onClose()` is called before/after navigate in DirectTripSheet — check ordering to avoid sheet re-open flicker
- All attendees are selected by default in TripSheet (line 94: `new Set(profiles.map((p) => p.uid))`) — but if the crew-load query fails, `selectedAttendees` is an empty Set and `attendees` would be `[]`

### Integration Points
- TripSheet `handleContinueToVoting` → `addDoc` → navigate to `/trips/${docRef.id}`
- TripDetail at `/trips/:id` with `status: 'proposed'` → shows "Ready to vote?" panel for creator
- Creator sees "Open voting" button → calls `openVoting(tripId)` → status becomes `'voting'`

</code_context>
