# Plan 05-04: Drive cache — incremental save and partial resume

**Phase:** 05 — Performance foundation
**Requirement IDs:** PERF-CACHE
**Executor:** gsd-executor
**Depends on:** 05-01 (Map.tsx `checkAndBuildCache` callsite is touched; complete Plan 05-01 first)

## Objective

Modify `buildDriveCache` to accept an existing partial cache and the user's `uid`, skip destinations already cached, and persist to Firestore after each batch of 5 completes. This closes a correctness bug: previously, an interrupted build (e.g., network drop at destination 63/70) would leave a partial cache that `isCacheValid`'s 90%-threshold check could accept as "valid", silently leaving up to 7 destinations uncached forever. With incremental saves and resume logic, a partial cache is detected and completed rather than accepted.

**Full rewrite of `buildDriveCache` for clarity:**
```ts
export async function buildDriveCache(
  homeLat: number,
  homeLng: number,
  uid: string,
  existingCache?: DriveCache,
  onProgress?: (done: number, total: number) => void
): Promise<DriveCache> {
  const cache: DriveCache = existingCache ? { ...existingCache } : {}
  const toFetch = destinations.filter((d) => !cache[d.id])
  const total = destinations.length
  let done = destinations.length - toFetch.length  // already-cached count

  for (let i = 0; i < toFetch.length; i += BATCH_SIZE) {
    const batch = toFetch.slice(i, i + BATCH_SIZE)
    await Promise.all(
      batch.map(async (dest) => {
        const result = await fetchOne(homeLat, homeLng, dest.lat, dest.lng)
        if (result) {
          cache[dest.id] = { ...result, cachedAt: Date.now() }
        }
        done++
        onProgress?.(done, total)
      })
    )
    await saveDriveCache(uid, cache, homeLat, homeLng)

    if (i + BATCH_SIZE < toFetch.length) {
      await sleep(BATCH_DELAY_MS)
    }
  }

  return cache
}
```

**Write count:** For a full fresh build of 70 destinations in batches of 5 = 14 batch saves. For a resumed build from 60/70 = 2 batch saves. This is the intended tradeoff (D-14).

**Full updated `checkAndBuildCache` block:**
```ts
async function checkAndBuildCache() {
  const snap = await getDoc(doc(db, 'users', currentUid!))
  if (!snap.exists()) return
  const profile = snap.data() as UserProfile
  const loc = profile.homeLocation
  if (!loc) return

  if (isCacheValid(profile.driveCache, profile.driveCacheLocation, loc.lat, loc.lng)) {
    setDriveCache(profile.driveCache ?? null)
    return
  }

  // Cache missing, stale, or partial — build/resume it
  setCacheProgress({ done: 0, total: destinations.length })
  try {
    const cache = await buildDriveCache(
      loc.lat,
      loc.lng,
      currentUid!,
      profile.driveCache,
      (done, total) => {
        setCacheProgress({ done, total })
      }
    )
    setDriveCache(cache)
  } catch (err) {
    console.error('Drive cache build failed:', err)
  } finally {
    setCacheProgress(null)
  }
}
```

**Verify:** On a fresh account (no driveCache), the Map page builds a new cache and the progress bar advances batch-by-batch. On a partially-built cache (manually set 60/70 entries in Firestore), the build resumes from the 61st destination rather than starting over.

| Threat | Severity | Mitigation |
|--------|----------|------------|
| Mid-batch save fails (Firestore write error) — build continues but progress is lost | Low | Firestore `updateDoc` error is not caught inside the batch loop — consider wrapping `saveDriveCache` in a try/catch that logs but does not throw, so the build continues |
| `saveDriveCache` called ~14 times per full build — Firestore write rate limit | Low | At 8 users, 14 writes per build per user is well within Firestore's 1-write/second-per-document limit (batches have 350ms delays between them) |
| Stale `existingCache` passed if user's home location changed since last build | Low | `isCacheValid` already returns `false` when location changes; `checkAndBuildCache` only passes `existingCache` after `isCacheValid` returns `false` — if location changed, the passed cache contains entries from the old location. | `toFetch` filter keeps old-location entries from `existingCache`, mixing location data | Med | If location changed, pass `undefined` instead of `profile.driveCache` to force a full rebuild.
