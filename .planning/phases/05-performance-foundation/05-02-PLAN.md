# Plan 05-02: Avatar migration — base64 Firestore blobs to Firebase Storage URLs

**Phase:** 05 — Performance foundation
**Requirement IDs:** PERF-AVATAR
**Executor:** gsd-executor
**Depends on:** 05-01 (CrewContext must exist; no file conflicts otherwise)

## Objective

Stop writing base64 data URIs (up to ~70 KB each) into Firestore user documents. New avatar uploads go to Firebase Storage at `avatars/{uid}.jpg`, and the download URL is written to `users/{uid}.customPhotoURL`. On profile load, any existing base64 value is detected and lazily migrated to Storage automatically — no admin script needed. `getUserPhoto()` requires no changes because it already returns `customPhotoURL` as-is.

## Pre-conditions

- [ ] Firebase project has a Storage bucket configured (`VITE_FIREBASE_STORAGE_BUCKET` is set in `.env`)
- [ ] `firebase@^10` is already installed (it includes `firebase/storage` in the bundle)
- [ ] `src/firebase.ts` currently does NOT export `storage` or `getStorage`

**Replace with Storage upload + URL write:**

1. Add imports at top of file:
   ```ts
   import { ref, uploadBytes, getDownloadURL } from 'firebase/storage'
   import { storage } from '../firebase'
   ```

2. Replace `handlePhotoChange`:
   ```ts
   async function handlePhotoChange(e: React.ChangeEvent<HTMLInputElement>) {
     const file = e.target.files?.[0]
     if (!file) return
     setPhotoUploading(true)
     try {
       const dataUri = await processImage(file)
       // Convert base64 data URI to Blob for Storage upload
       const res = await fetch(dataUri)
       const blob = await res.blob()
       const storageRef = ref(storage, `avatars/${profile.uid}.jpg`)
       await uploadBytes(storageRef, blob, { contentType: 'image/jpeg' })
       const downloadURL = await getDownloadURL(storageRef)
       await updateDoc(doc(db, 'users', profile.uid), { customPhotoURL: downloadURL })
       setCustomPhoto(downloadURL)
     } catch (err) {
       console.error('Photo upload failed:', err)
     } finally {
       setPhotoUploading(false)
       if (fileInputRef.current) fileInputRef.current.value = ''
     }
   }
   ```

The `processImage` function (lines 55–93) is unchanged — it still produces a compressed JPEG data URI at 128×128px. The change is only in what is done with that data URI afterwards.

### Task 3: Add lazy migration for existing base64 avatars (D-07)

**File:** `src/pages/Profile.tsx`

On profile load, check if the current `customPhotoURL` is a legacy base64 data URI. If so, perform the Storage upload + URL overwrite once and update local state. This runs transparently on first profile load for any user with an old base64 photo.

Add a `useEffect` near the top of `ProfilePage` (after the existing state initialisations), after the `storage` import is in place:

```ts
// Lazy migration: if customPhotoURL is a legacy base64 blob, upload it to Storage
useEffect(() => {
  if (!profile.customPhotoURL?.startsWith('data:')) return

  async function migrateAvatar() {
    try {
      const res = await fetch(profile.customPhotoURL!)
      const blob = await res.blob()
      const storageRef = ref(storage, `avatars/${profile.uid}.jpg`)
      await uploadBytes(storageRef, blob, { contentType: 'image/jpeg' })
      const downloadURL = await getDownloadURL(storageRef)
      await updateDoc(doc(db, 'users', profile.uid), { customPhotoURL: downloadURL })
      setCustomPhoto(downloadURL)
    } catch (err) {
      console.error('Avatar migration failed — will retry on next load:', err)
    }
  }

  migrateAvatar()
}, []) // eslint-disable-line react-hooks/exhaustive-deps
  // Intentionally runs once on mount; profile.customPhotoURL is a stable prop
```

**Verify:** With a test user that has a base64 `customPhotoURL`, load the Profile page. After a moment, Firestore `users/{uid}.customPhotoURL` updates to an HTTPS URL. Avatar still displays correctly throughout.

## Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| Storage bucket not configured (missing `storageBucket` in firebaseConfig) | High | Pre-condition check — `VITE_FIREBASE_STORAGE_BUCKET` must be set in `.env`; upload will throw a clear error if missing |
| Unauthenticated write to `avatars/` Storage path | High | Firebase Storage security rules must restrict writes to `request.auth.uid == {uid}` — verify rules in Firebase Console before testing |
| Lazy migration re-runs on every profile load if upload succeeds but Firestore write fails | Low | If Firestore write fails, `customPhotoURL` remains `data:` prefix; migration retries next load — this is acceptable and self-healing |
| Large base64 blob in `fetch()` during lazy migration fails on slow connections | Low | Failure is caught and logged; original base64 photo still displays via `customPhoto` state |
| Storage download URLs expire | Low | Firebase Storage URLs do not expire by default for public objects; if using signed URLs, they would — use `getDownloadURL()` which produces permanent public URLs |
</threat_model>
